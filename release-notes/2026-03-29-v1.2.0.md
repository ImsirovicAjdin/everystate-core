# @everystate/core v1.2.0

**Date:** 2026-03-29
**Previous:** v1.1.0

## Summary

Fixes a batch flush consistency bug where subscribers on early paths in a batch
saw stale values for later paths. The `flushBatch()` internals are now two-pass:
all values are written to state first, then all subscribers are notified.

This is a **bugfix release**. No API changes. Drop-in replacement for v1.1.0.

## The bug

When `batch()` or `setMany()` grouped multiple `set()` calls, the flush would
write and notify one entry at a time:

```js
// v1.1.0 flushBatch (broken)
for (const [p, v] of entries) {
  writeAndNotify(p, v);  // writes value AND fires subscribers before next entry
}
```

A subscriber on the **first** path set in the batch would fire before later
paths had been written. Any `store.get()` call inside that subscriber for a
later path would return the **stale** value:

```js
store.subscribe('toggle.active', () => {
  store.get('color');  // ← still 'gray', not 'blue'!
});

store.batch(() => {
  store.set('toggle.active', true);  // subscriber fires HERE
  store.set('color', 'blue');        // not yet written
});
```

This caused the "one click behind" bug in the typed-array toggle demo, where
the DOM would update with stale values from the previous state.

## The fix

`flushBatch()` is now two-pass:

```js
// v1.2.0 flushBatch (fixed)
// Pass 1: write all values to state
const written = [];
for (const [p, v] of entries) {
  written.push(writeToState(p, v));
}
// Pass 2: fire all notifications (all values now fresh)
for (const entry of written) {
  notifyListeners(entry);
}
```

Internally, `writeAndNotify` was split into two functions:
- `writeToState(path, value)` — writes to state, returns entry metadata
- `notifyListeners(entry)` — fires exact, wildcard, and global subscribers

Non-batch `set()` calls still use the combined `writeAndNotify()` (no behavior
change for non-batched writes).

## Before/After

### v1.1.0: subscriber sees stale sibling values

```js
const store = createEveryState({ active: false, color: 'gray' });

store.subscribe('active', () => {
  console.log(store.get('color')); // 'gray' ← STALE
});

store.batch(() => {
  store.set('active', true);
  store.set('color', 'blue');
});
```

### v1.2.0: subscriber sees all fresh values

```js
const store = createEveryState({ active: false, color: 'gray' });

store.subscribe('active', () => {
  console.log(store.get('color')); // 'blue' ← FRESH
});

store.batch(() => {
  store.set('active', true);
  store.set('color', 'blue');
});
```

## What changed

| | v1.1.0 | v1.2.0 |
|---|---|---|
| **`flushBatch` passes** | 1 (write+notify interleaved) | 2 (write-all, then notify-all) |
| **Subscriber reads during batch flush** | May see stale sibling values | Always sees fresh values |
| **`sync()` pattern** | Must subscribe to last path set | Can subscribe to any path |
| **Non-batch `set()` behavior** | Unchanged | Unchanged |
| **API surface** | Unchanged | Unchanged |

The core difference: in v1.1.0, batch flush order leaked into subscriber
behavior. In v1.2.0, subscribers always see a consistent snapshot regardless
of which path they subscribe to.
